# Task Specification: T23 - Sprint 2

## Objective
Add short-lived memoization for git status results keyed by repository and tracking scope to avoid redundant expensive polling work.

## Files to Modify
- `src/services/git-status-service.ts`: Add optional memoization layer keyed by scope/branch context.
- `src/server/activity-cache-service.ts`: Integrate invalidation semantics with memoized results.
- `tests/backend/services/git-status-service.test.ts`: Add memoization/invalidation test coverage.

## Technical Details & Research Findings
- **Context**: frequent status polling can repeatedly execute equivalent GH queries.
- **Goal**: reduce redundant calls while keeping freshness constraints explicit.
- **Constraint**: memoization must be short-lived and invalidatable by status-changing actions.

## Execution Steps
1. Implement cache key model that includes repo path and tracking request parameters.
2. Apply TTL-based memoization in status fetch path.
3. Expose/trigger invalidation for state-changing operations and runner updates.
4. Add tests for cache hit, miss, TTL expiry, and invalidation.

## Verification Requirements
- **Automated Tests**: `npm run test -- tests/backend/services/git-status-service.test.ts`; `npm run test -- tests/backend/server/dashboard-server.test.ts`
- **Criteria**: Equivalent status requests reuse cached data within TTL window.
- **Criteria**: Cache invalidates correctly after relevant state changes.

## Engineering Standard
- Use the feature branch: feature/sprint2-implementation
- Ensure all tests pass before completing.
