# Task Specification: T24 - Sprint 1

## Objective
Implement a robust, shared subprocess command runner in `src/shared/subprocess/command-runner.ts` and standardize the execution and error contracts across the project.

## Files to Modify
- `src/shared/subprocess/command-runner.ts`: Create this new file.
- `src/services/cli-process-runner.ts`: Refactor to use the new common runner.
- `src/services/git-status-service.ts`: Migrate to the new common runner.

## Technical Details & Research Findings
- **Context**: Subprocess execution currently uses `spawn` in a slightly different way in `cli-process-runner.ts` and other places. Error reporting is often ad-hoc and inconsistent.
- **Common Runner Design**:
    - Implement a `CommandRunner` with support for streaming output (via callbacks), timeouts, and environment configuration.
    - Provide a standardized `CommandResult` that includes exit code, stdout, and clipped stderr.
    - Implement robust error handling for common issues (e.g., EADDRINUSE, ENOENT, timeouts).
- **Constraints**: Ensure the runner is fully asynchronous and supports capturing both stdout and stderr reliably.

## Execution Steps
1. Create `src/shared/subprocess/command-runner.ts`.
2. Move and generalize the `runStreamingCommand` logic.
3. Add support for optional execution timeouts and normalized stderr clipping for log clarity.
4. Refactor `cli-process-runner.ts` to delegate to the new `CommandRunner`.
5. Refactor `src/services/git-status-service.ts` to use the standardized runner instead of manual `exec` or `spawn` calls.
6. Verify with unit tests in `tests/backend/shared/subprocess/command-runner.test.ts`.

## Verification Requirements
- **Automated Tests**: Run `npm test tests/backend/shared/subprocess/command-runner.test.ts`.
- **Criteria**: Command execution behaves consistently across various scenarios, including successes, failures, and timeouts.
