# Task Specification: T11 - Sprint 1

## Objective
Implement a strongly typed MCP tool registry in `src/api/mcp/tool-registry.ts` to replace the current `any`-based handler map in `src/server/mcp-request-router.ts`.

## Files to Modify
- `src/api/mcp/tool-registry.ts`: Create this new file.
- `src/server/mcp-request-router.ts`: Migrate to use the new registry.

## Technical Details & Research Findings
- **Context**: The `handlers` object in `mcp-request-router.ts` uses `any` for several inputs (e.g., `create_session`, `task_agent`), which is fragile and bypasses type checking.
- **Tool Registry Design**:
    - Use Zod or a similar schema validator (or strict interfaces) to define the input/output of each tool.
    - Create a central registry that maps tool names to their respective handlers with full type safety.
    - Implement a `register` method and a `dispatch` method that leverages the type information.
- **Constraints**: Ensure the registry is fully compatible with the MCP SDK's tool definitions.

## Execution Steps
1. Create `src/api/mcp/tool-registry.ts`.
2. Define a complete set of interfaces or Zod schemas for all tool inputs (matching the definitions in `mcp-tool-definitions.ts`).
3. Refactor `registerMcpRequestHandlers` to use the typed registry instead of an inline object.
4. Replace `any` payloads with their respective typed interfaces.
5. Verify that the project still compiles and that the MCP tools are correctly dispatched.

## Verification Requirements
- **Automated Tests**: Run `npm test tests/backend/mcp/tool-registry.test.ts` (create if missing).
- **Criteria**: TypeScript compiler correctly identifies mismatched payloads for registered tools.

## Engineering Standard
- Use the feature branch: feature/sprint1-implementation
- Ensure all tests pass before completing.
