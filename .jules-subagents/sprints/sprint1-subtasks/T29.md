# Task Specification: T29 - Sprint 1

## Objective
Implement a centralized polling manager hook in the dashboard to unify the runtime status and git tracking polling loops, and add support for exponential backoff and explicit refresh triggers.

## Files to Modify
- `dashboard/src/hooks/use-dashboard-poll-manager.ts`: Create this new file.
- `dashboard/src/hooks/use-dashboard-runtime-data.ts`: Refactor to use the new manager.

## Technical Details & Research Findings
- **Context**: `use-dashboard-runtime-data.ts` currently manages two independent `setInterval` loops for runtime status and git status. This can lead to race conditions and inefficient polling when the server is unresponsive.
- **Polling Manager Design**:
    - Implement a `useDashboardPollManager` hook that handles the polling lifecycle for multiple data sources.
    - Support features like: configurable interval, exponential backoff on errors, and a `refreshNow` function to manually trigger updates.
    - Centralize the unified error state and connection status.
- **Constraints**: Maintain the existing `refreshRuntimeStatus` and `refreshGitStatus` functionality.

## Execution Steps
1. Create `dashboard/src/hooks/use-dashboard-poll-manager.ts`.
2. Implement the polling logic with support for backoff and refresh triggers.
3. Refactor `use-dashboard-runtime-data.ts` to delegate its polling responsibilities to the new manager.
4. Ensure that the dashboard still correctly displays connection errors and status updates.
5. Verify that manual refreshes (e.g., after saving settings) correctly reset the polling timers.

## Verification Requirements
- **Manual Check**: Observe the dashboard's network activity in the browser to confirm that polling is consistent and handles backend downtime gracefully.
- **Criteria**: Dashboard polling is more robust and avoids duplicate intervals or race conditions.
