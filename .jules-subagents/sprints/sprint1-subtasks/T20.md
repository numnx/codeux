# Task Specification: T20 - Sprint 1

## Objective
Optimize the performance of `runSessionSyncStep` by pre-indexing sessions by their task run key, reducing the time complexity from O(tasks * sessions) to O(tasks + sessions).

## Files to Modify
- `src/sprint/steps/session-sync-step.ts`: Refactor the session lookup logic.

## Technical Details & Research Findings
- **Context**: The `runSessionSyncStep` currently performs a linear scan of all sessions for every task in the sprint to find a matching session by its title. As the number of tasks and total sessions grows, this becomes a significant bottleneck.
- **Optimization Strategy**:
    - Iterate through all fetched sessions once and build a `Map<string, JulesSession>` where the key is the extracted `taskRunKey`.
    - Handle cases where multiple sessions might match the same run key (e.g., retried tasks) by sorting by `createTime` during pre-indexing.
    - Change the task loop to perform a constant-time lookup in the map.
- **Constraints**: Ensure the correct (most recent) session is always picked for each task.

## Execution Steps
1. Refactor `runSessionSyncStep` to pre-index the `sessions` array into a `Map`.
2. Extract the run key from each session's title only once.
3. Replace the nested `sessions.find` call with a direct `map.get` call.
4. Add a benchmark test or a high-volume test case in `tests/backend/sprint/session-sync-step.test.ts` to verify the performance improvement.
5. Verify that the session synchronization logic remains correct for various task states.

## Verification Requirements
- **Automated Tests**: Run `npm test tests/backend/sprint/session-sync-step.test.ts`.
- **Criteria**: Execution time for synchronization with a large number of sessions (e.g., 500+) is significantly reduced while maintaining correctness.
