# Task Specification: T08 - Sprint 1

## Objective
Implement a structured logging utility in `src/shared/logging/logger.ts` and add support for correlation IDs to track requests across the MCP server and dashboard API.

## Files to Modify
- `src/shared/logging/logger.ts`: Create this new file.
- `src/shared/logging/correlation-id.ts`: Create this new file.
- `src/app/dependency-factory.ts`: Inject the logger into services.

## Technical Details & Research Findings
- **Context**: The codebase currently uses raw `console` methods, which makes log filtering and request correlation difficult.
- **Logger Requirements**:
    - Support log levels: `debug`, `info`, `warn`, `error`.
    - Support structured metadata (objects).
    - Output format: JSON in production, human-readable in development.
    - Include `correlationId` in every log message if present.
- **Correlation ID Requirements**:
    - Utility to generate/get correlation IDs from `AsyncLocalStorage` or equivalent.
    - Middleware for Express to seed IDs.
    - Logic to pass IDs to MCP tool handlers.

## Execution Steps
1. Create `src/shared/logging/logger.ts` and `src/shared/logging/correlation-id.ts`.
2. Implement a simple, dependency-free structured logger.
3. Integrate `AsyncLocalStorage` to store and retrieve the current correlation ID.
4. Update `dependency-factory.ts` to provide a logger instance to all services.
5. Add middleware to `src/server/dashboard-server.ts` to assign correlation IDs to incoming HTTP requests.
6. Seed correlation IDs into MCP tool execution contexts in `src/server/jules-agent-server.ts`.

## Verification Requirements
- **Automated Tests**: Run `npm test tests/backend/shared/logging/logger.test.ts`.
- **Criteria**: Logs are emitted with consistent structure and include correlation IDs when available.

## Engineering Standard
- Use the feature branch: feature/sprint1-implementation
- Ensure all tests pass before completing.
