# Task Specification: T26 - Sprint 1

## Objective
Unify the `DashboardSettings` and related types into a single shared contract in `src/contracts/app-types.ts` and ensure both backend and dashboard builds consume this canonical source.

## Files to Modify
- `src/contracts/app-types.ts`: Primary source of truth.
- `dashboard/src/types.ts`: Remove duplicated types and import from shared location.
- `vite.config.ts`: Update to handle shared imports if necessary.

## Technical Details & Research Findings
- **Context**: `DashboardSettings` and its nested interfaces (Git, CI, AI Provider, etc.) are currently duplicated in `src/contracts/app-types.ts` and `dashboard/src/types.ts`.
- **Improvements**:
    - Centralize all settings-related interfaces in `src/contracts/app-types.ts`.
    - Centralize default settings values (from `src/repositories/settings-defaults.ts`) so they can be reused by the dashboard.
- **Constraints**: Maintain compatibility with the Preact/Vite build for the dashboard.

## Execution Steps
1. Audit both type files to ensure all fields are present in the shared `app-types.ts`.
2. Update `dashboard/src/types.ts` to export types from `../../src/contracts/app-types.js` (or use a path alias).
3. Ensure that constants like `DEFAULT_DASHBOARD_SETTINGS` are also shared or easily accessible.
4. Verify that both the backend and dashboard projects still compile without type errors.
5. Check for any naming inconsistencies between the two layers (e.g., `TaskStatus` vs `SubtaskStatus`).

## Verification Requirements
- **Automated Tests**: Run `npm run build` for both root and dashboard.
- **Criteria**: No type errors in either build, and settings correctly serialize/deserialize between the layers.

## Engineering Standard
- Use the feature branch: feature/sprint1-implementation
- Ensure all tests pass before completing.
