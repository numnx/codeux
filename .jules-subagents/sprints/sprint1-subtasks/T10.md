# Task Specification: T10 - Sprint 1

## Objective
Migrate `src/server/jules-agent-server.ts`, `src/sprint/sprint-orchestrator.ts`, `src/services/cli-workflow-service.ts`, and `src/server/dashboard-server.ts` to use the new structured logger instead of raw `console` methods.

## Files to Modify
- `src/server/jules-agent-server.ts`: Replace `console.error` and `console.log`.
- `src/sprint/sprint-orchestrator.ts`: Replace `console.error` and `console.log`.
- `src/services/cli-workflow-service.ts`: Replace `console.error` and `console.log`.
- `src/server/dashboard-server.ts`: Replace `console.error` and `console.log`.

## Technical Details & Research Findings
- **Context**: Centralized logging is a prerequisite for better observability and troubleshooting.
- **Logger Usage**:
    - Inject the logger through the `deps` object in each service.
    - Use appropriate log levels (e.g., `info` for startup, `error` for failures).
    - Attach relevant context as objects (e.g., `sessionId`, `taskId`).
- **Constraints**: Do not remove the bootstrap fatal error handler in `src/index.ts`, which may still need `console.error`.

## Execution Steps
1. Update service dependencies to include the logger.
2. Systematically replace each `console.log` and `console.error` call with the logger.
3. Ensure structured metadata is used where appropriate (e.g., when logging objects or errors).
4. Verify that logs are correctly formatted and include the expected context and correlation IDs.

## Verification Requirements
- **Automated Tests**: Run existing tests for these services and check that they don't crash due to logger injection.
- **Criteria**: No direct `console` usage remains in the target files.

- Use the feature branch: feature/sprint1-implementation
