# Task Specification: T04 - Sprint 1

## Objective
Refactor `src/config/app-config.ts` to utilize the new shared search path resolver and config value readers, and decompose it into cleaner, more testable components.

## Files to Modify
- `src/config/app-config.ts`: Substantially refactor the `loadAppConfig` logic.

## Technical Details & Research Findings
- **Context**: `loadAppConfig` currently mixes argument parsing, environment variable reading, and file-based config loading for the API key and dashboard port.
- **Decomposition Target**:
    - Extract pure CLI argument parsing for `--api-key` and `--help`.
    - Extract specific loaders: `apiKeyLoader` (env -> config.json -> settings.json) and `dashboardPortLoader`.
    - Use `buildCandidatePaths` from `src/shared/config/search-paths.ts`.
    - Use readers from `src/shared/config/value-readers.ts`.
- **Constraints**: Maintain current precedence: CLI arg > Env var > `.jules-subagents/settings.json` (for API key) or `config.json` (for Port).

## Execution Steps
1. Update `src/config/app-config.ts` imports to include the new shared utilities.
2. Refactor `parseApiKeyArg` and `loadAppConfig` to use `buildCandidatePaths`.
3. Migrate `readPortValue` usage to `src/shared/config/value-readers.ts`.
4. Ensure the server's startup flow is preserved and correctly logs errors for invalid configs.
5. Expand unit tests in `tests/backend/config/app-config.test.ts` to cover the decomposed loaders.

## Verification Requirements
- **Automated Tests**: Run `npm test tests/backend/config/app-config.test.ts`.
- **Criteria**: All tests pass, and configuration values are correctly resolved from all supported sources.
