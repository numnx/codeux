# Task Specification: T15 - Sprint 1

## Objective
Refactor the `SprintOrchestrator` by decomposing the monolithic `execute` and `runOrchestrationCycle` methods into smaller, specialized runner classes.

## Files to Modify
- `src/domain/sprint/orchestrator/cycle-runner.ts`: Create this new file.
- `src/domain/sprint/orchestrator/watch-loop-runner.ts`: Create this new file.
- `src/sprint/sprint-orchestrator.ts`: Refactor to delegate to new runners.

## Technical Details & Research Findings
- **Context**: The `SprintOrchestrator` class is nearly 900 lines long and handles cycle logic, watch mode, CI gating, and status reporting in one place.
- **Decomposition Target**:
    - `CycleRunner`: Handles a single orchestration pass (sync, derivation, start, intervention, protocol).
    - `WatchLoopRunner`: Handles the continuous execution loop with sleep and termination logic.
    - `SprintOrchestrator`: Becomes a high-level facade that coordinates these runners.
- **Constraints**: Maintain existing step-based settings and dependency injection patterns.

## Execution Steps
1. Create `src/domain/sprint/orchestrator/cycle-runner.ts` and move the core logic of `runOrchestrationCycle` into it.
2. Create `src/domain/sprint/orchestrator/watch-loop-runner.ts` and move the `while` loop logic from `execute` into it.
3. Refactor `SprintOrchestrator` to instantiate and call these new runners.
4. Update unit tests in `tests/backend/sprint/sprint-orchestrator.test.ts` to reflect the new structure.
5. Ensure the `SprintCycleResult` and `SprintAgentArgs` remain compatible.

## Verification Requirements
- **Automated Tests**: Run `npm test tests/backend/sprint/sprint-orchestrator.test.ts`.
- **Criteria**: Sprint orchestration behaves identically in both single-run and watch modes.
