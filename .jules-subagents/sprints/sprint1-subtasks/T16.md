# Task Specification: T16 - Sprint 1

## Objective
Extract the feature branch CI gating logic from `SprintOrchestrator` into a dedicated domain service `src/domain/sprint/ci/feature-pr-gate.ts`.

## Files to Modify
- `src/domain/sprint/ci/feature-pr-gate.ts`: Create this new file.
- `src/sprint/sprint-orchestrator.ts`: Refactor to delegate CI gating to the new service.

## Technical Details & Research Findings
- **Context**: `applyFeatureBranchCiGate` contains complex logic for monitoring pull requests, handling CI failures, and triggering autofixes.
- **Decomposition Target**:
    - Move logic for identifying completed tasks awaiting merge and evaluating CI status.
    - Move logic for notifying Jules about failed CI (autofix triggering).
    - Provide a clean interface: `evaluateCiGate(subtasks: Subtask[], context: CiGateContext): Promise<CiGateResult>`.
- **Constraints**: Service should be purely logic-based and receive necessary state (e.g., `GitTrackingStatus`) as input.

## Execution Steps
1. Create `src/domain/sprint/ci/feature-pr-gate.ts`.
2. Move `applyFeatureBranchCiGate` and `notifyJulesAboutFailedCi` logic to the new service.
3. Update `SprintOrchestrator` to call `FeaturePrGateService.evaluateCiGate` during the orchestration cycle.
4. Define clear input/output interfaces to decouple the gate service from the orchestrator's internal state.
5. Add unit tests for `FeaturePrGateService` covering various PR and CI status combinations.

## Verification Requirements
- **Automated Tests**: Run `npm test tests/backend/domain/sprint/ci/feature-pr-gate.test.ts` (create if missing).
- **Criteria**: CI gating, autofix triggering, and status reporting logic remains correct and well-isolated.
