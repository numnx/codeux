# Task Specification: T27 - Sprint 1

## Objective
Implement reusable, immutable settings update helpers in `dashboard/src/lib/settings-updaters.ts` to simplify nested state mutations in dashboard components.

## Files to Modify
- `dashboard/src/lib/settings-updaters.ts`: Create this new file.
- `dashboard/src/components/settings/*.tsx`: Refactor to use the new helpers.

## Technical Details & Research Findings
- **Context**: Dashboard settings components currently use deeply nested object spread syntax (e.g., `{ ...settings, aiProvider: { ...settings.aiProvider, providers: { ... } } }`) for every input change, which is error-prone and hard to read.
- **Helper Design**:
    - Provide a set of specialized updater functions: `updateAiProvider(settings, patch)`, `updateGitSettings(settings, patch)`, `updateProviderConfig(settings, providerId, patch)`, etc.
    - Use a consistent functional pattern that returns a new `DashboardSettings` object.
- **Constraints**: Helpers must be strictly typed and ensure no accidental state mutation.

## Execution Steps
1. Create `dashboard/src/lib/settings-updaters.ts`.
2. Implement typed updater functions for each major settings section (AI Provider, Git, CI, Loop, Workflow).
3. Refactor `AiProviderSection.tsx`, `GitSettingsSection.tsx`, `CiIntelligenceSection.tsx`, and `CliWorkflowSection.tsx` to call these helpers.
4. Verify that settings changes still propagate correctly to the `onChange` callback and the backend.

## Verification Requirements
- **Manual Check**: Interact with various settings toggles and inputs in the dashboard and ensure they update correctly and persist on save.
- **Criteria**: Component code is cleaner and avoids manual nested object reconstruction.

## Engineering Standard
- Use the feature branch: feature/sprint1-implementation
- Ensure all tests pass before completing.
