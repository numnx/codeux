## 🔍 Key Findings & Observations

1. **Unused Space Fragmentation (Freelist)**: Prior to `VACUUM`, `app.db` and `settings.db` held a combined **~82 MB of stale pages** (freelist) on disk. These are caused by updates/deletions that SQLite keeps reserved rather than releasing to the OS.
2. **Granular Log Accumulation**: The vast majority of records (~85%) are generated by low-level, high-frequency execution history tables (`task_run_events` and `provider_activities`). These represent step-by-step logs of previous agent executions.
3. **No Automated Pruning**: Currently, there is **no pruning code or lifecycle policy** implemented for completed task runs, expired sessions, or transient dashboard events. They accumulate indefinitely.
4. **Cascading Foreign Keys**: Fortunately, the schema uses `ON DELETE CASCADE` extensively. For example, deleting a task run from `task_runs` automatically cleans up all associated rows in `task_run_events`, `provider_invocations`, `execution_invocations`, and `execution_invocation_messages`.

## 🛠️ Optimization Options

Here are the best options we have to keep database growth under control:

### Option 1: Automated Retention-Based Pruning Service (Recommended)
Introduce a new service, [DatabasePruneService](file:///home/pierre/jules-agent-mcp/src/services/database-prune-service.ts), instantiated inside [core-factory.ts](file:///home/pierre/jules-agent-mcp/src/app/dependency-factory/core-factory.ts) and run during startup (alongside `DockerAssetPruneService`).

- **Mechanism**:
  - Delete completed `task_runs` older than $N$ days (e.g. 14 days). This automatically cascades to clean up the heaviest tables: `task_run_events` and `execution_invocation_messages`.
  - Delete completed/failed `provider_sessions` (and their un-cascaded `provider_activities` logs) older than $N$ days.
  - Delete resolved `project_attention_items` older than $N$ days.
  - Delete stale `dashboard_realtime_events` older than 1 day.
  - Execute a database `VACUUM` on completion to release free space to the OS.
- **Pros**: Zero-touch, runs automatically, keeps databases compact over time.
- **Cons**: Retention period needs to be carefully chosen (e.g. 14 days by default) so users don't lose recent log history.

### Option 2: Expose a CLI command / MCP Tool for Pruning
Expose the pruning functionality directly to the user (via CLI flag like `codeux database prune --days 7`) or as an MCP tool (e.g., `prune_database`).

- **Pros**: Gives full control to the user to choose when to optimize and what retention period to apply.
- **Cons**: Requires manual trigger; the database can still grow unchecked if the user forgets to run it.

### Option 3: Enable Incremental Auto-Vacuum
Configure SQLite to use incremental auto-vacuum:
```sql
PRAGMA auto_vacuum = INCREMENTAL;
PRAGMA incremental_vacuum(200); -- vacuum up to 200 pages at a time
```
- **Pros**: Automatically shrinks the database file gradually.
- **Cons**: Increases write overhead and database fragmentation, and still requires rows to be deleted first (does not solve the retention/bloat issue itself).
